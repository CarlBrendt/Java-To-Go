#### 1. Модель Состояния (State Schema)
В LangGraph мы определим единый объект состояния (`GlobalState`), который будет передаваться между узлами. Он будет содержать:
*   **`java_source_code`**: Исходный код Java (разбитый на файлы или сущности).
*   **`api_contract`**: Извлеченный контракт (OpenAPI/Swagger) из Java-сервиса.
*   **`go_target_structure`**: Предложенная структура проекта Go (модуль, пакеты, файлы).
*   **`migration_plan`**: Пошаговый план переноса (что переносим, что рефакторим).
*   **`intermediate_representations`**: Промежуточные модели (DTO, бизнес-логика, слои).
*   **`generated_go_code`**: Черновики кода на разных этапах.
*   **`test_results`**: Результаты сравнительного тестирования (Java vs Go).
*   **`human_feedback`**: Заметки и правки от инженера на каждом этапе.
*   **`final_report`**: Итоговый отчет для жюри.
*   **`status`**: Текущее состояние пайплайна (анализ, генерация, тесты, готово).

### 2. Архитектура Пайплайна (Steps & Nodes)

Пайплайн состоит из 7 ключевых этапов, организованных в граф.

#### Этап 1: Deep Analysis & Contract Extraction (Анализ)
*   **Цель:** Понять структуру проекта без использования регулярных выражений.
*   **Действия:**
    *   Парсинг Java-кода с использованием AST (Abstract Syntax Tree) или специализированных парсеров (например, `javaparser` через subprocess или встроенные библиотеки).
    *   Извлечение всех Endpoints (Controller-классы, `@RequestMapping`, `@PostMapping` и т.д.).
    *   Извлечение DTO и моделей данных.
    *   Формирование **JSON-карты API** (метод, путь, тело запроса, тело ответа, статусы).
*   **Роль AI:** Анализировать AST, предлагать названия пакетов в Go, выделять сложные зависимости.
*   **Выход:** Структурированный `api_contract` и `java_structure_map`.

#### Этап 2: Strategic Planning & Refactoring Map (Планирование)
*   **Цель:** Определить стратегию переноса.
*   **Действия:**
    *   Сравнение Spring Boot паттернов (Dependency Injection, Annotations) с Go-аналогами (Interfaces, Structs).
    *   Определение, какие библиотеки Spring (JPA, Security, Validation) нужно заменить на Go-эквиваленты (GORM, Gorilla/mux, Validator).
    *   Генерация **Migration Plan** (список файлов, которые нужно создать, и логика, которую нужно перенести).
*   **Точка контроля (Human-in-the-loop):** AI предлагает план. Инженер может отредактировать его (например, "использовать стандартную библиотеку http вместо фреймворка", "изменить подход к логированию").
*   **Выход:** Утвержденный `migration_plan`.

#### Этап 3: Data Layer Transformation (Преобразование структур данных)
*   **Цель:** Перевести DTO и модели БД.
*   **Действия:**
    *   Конвертация Java-классов в Go-структуры.
    *   Обработка аннотаций валидации (`@NotNull`, `@Size`) в Go-теги (`json:"...", validate:"required"`).
    *   Обработка сериализации/десериализации (Jackson -> JSON/MsgPack).
*   **Роль AI:** Генерация кода структур, проверка на потерю типов (например, `Optional` -> `pointer` или `zero value`).
*   **Выход:** Набор Go-структур (`models.go`).

#### Этап 4: Business Logic Migration (Ядро логики)
*   **Цель:** Перенос сервисов и репозиториев.
*   **Действия:**
    *   Перевод сервисных слоев (Service Layer).
    *   Замена Spring Bean-инъекций на явную передачу зависимостей (Dependency Injection via struct embedding или constructor injection).
    *   Конвертация SQL-запросов (JPA/Hibernate) в SQL-запросы (GORM/sqlx) или Go-логику.
*   **Точка контроля:** AI генерирует черновик. Система автоматически проверяет наличие синтаксических ошибок и логических дыр (например, обработка nil-указателей).
*   **Выход:** Черновик бизнес-логики (`service.go`, `repository.go`).

#### Этап 5: API Layer & Routing (Слой API)
*   **Цель:** Создание Go-сервера, соответствующего контракту.
*   **Действия:**
    *   Генерация роутинга (Gin/Echo/Fiber) на основе `api_contract` из Этапа 1.
    *   Написание обработчиков (Handlers), которые вызывают бизнес-логику из Этапа 4.
    *   Настройка маппинга JSON (request -> struct, struct -> response).
*   **Роль AI:** Генерация шаблонного кода для handlers, настройка middleware (CORS, Logging).
*   **Выход:** `handlers.go`, `router.go`, `main.go`.

#### Этап 6: Verification & Testing (Валидация)
*   **Цель:** Доказать, что Go-сервис ведет себя так же, как Java.
*   **Действия:**
    *   **Сравнение контрактов:** Автоматическая проверка, что все эндпоинты Java есть в Go и имеют те же сигнатуры.
    *   **Генерация тестов:** Создание Go-тестов на основе сценариев из Java (если есть) или генерация тестов на граничных значениях.
    *   **Сравнительное тестирование (E2E):** Запуск обоих сервисов (или эмуляция) с набором тестовых данных и сравнение ответов.
*   **Точка контроля:** Если тесты не пройдены, граф возвращается на Этап 4 или 5 для исправления (Loop).
*   **Выход:** `test_report.json`, `diff_analysis`.

#### Этап 7: Reporting & Packaging (Финализация)
*   **Цель:** Собрать артефакты для жюри.
*   **Действия:**
    *   Генерация итогового отчета (Markdown/JSON): что сделано, где ручное вмешательство, где AI, какие ограничения.
    *   Формирование `Dockerfile` для Go-сервиса.
    *   Сборка проекта.
*   **Выход:** Финальный репозиторий, отчет.
